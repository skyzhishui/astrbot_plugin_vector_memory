# AstrBot 向量记忆插件 🧠

一个基于向量语义检索的长期记忆系统，让 AI 能够真正"记住"重要信息。

## ✨ 功能特点

- 🔍 **语义检索**: 使用向量相似度搜索，而非关键词匹配
- 💾 **自动记忆**: 自动从对话中提取重要信息并存储
- 🎯 **智能注入**: 在对话时自动检索相关记忆并注入上下文
- 📊 **分类管理**: 支持多种记忆类别（偏好、事实、事件等）
- ⭐ **重要性评分**: 每条记忆都有重要性权重
- 🔐 **记忆分层**: 支持公共记忆、用户专属记忆、秘密记忆三种可见性
- 🏷️ **所有者标识**: 每条记忆都记录所有者信息，方便追溯
- 🚫 **智能去重**: 自动检测并跳过语义相似度超过95%的重复记忆
- ⚡ **Embedding缓存**: LRU缓存机制，加速重复查询，减少API调用
- 🚀 **ChromaDB 驱动**: 使用 ChromaDB 实现高效的向量索引和检索
- 🎯 **分层架构**: L0/L1/L2/L3 四层记忆架构，优化 Token 消耗

## 🆕 v1.4.0 新特性：分层记忆架构

### 分层架构说明

| 层级 | 名称 | 加载策略 | 适用场景 | 预期Token |
|------|------|----------|----------|-----------|
| **L0** | 核心层 | 始终加载 | 称呼规则、安全偏好、身份映射 | ~200 |
| **L1** | 高频层 | 关键词触发 | 家庭成员、重要日期、常用配置 | ~150 |
| **L2** | 普通层 | 向量检索 | 开发规范、配置信息、一般记忆 | ~100 |
| **L3** | 归档层 | 明确询问 | 历史记录、低重要性信息 | 按需 |

### 预期效果

| 场景 | 优化前Token | 优化后Token | 节省 |
|------|-------------|-------------|------|
| 日常对话 | 1500-3000 | 200-450 | **60-80%** |
| 开发相关 | 1500-3000 | 350-600 | **70-80%** |

缓存命中率: 0% → **60-70%**

### 关键词触发机制

L1 层记忆支持关键词触发预加载：

- **家人/孩子** → 触发家庭成员相关信息
- **生日/纪念日** → 触发重要日期信息
- **ComfyUI/绘图/画图** → 触发 ComfyUI 配置
- **Sonos/音箱/语音** → 触发 Sonos 配置

## 📦 安装

1. 将插件放入 `AstrBot/data/plugins/astrbot_plugin_vector_memory/` 目录
2. 安装依赖: `pip install chromadb`
3. 在 AstrBot 管理面板中配置 Embedding Provider ID
4. 重启 AstrBot 或使用 `/memory_init` 初始化

## ⚙️ 配置说明

在 `_conf_schema.json` 中配置：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `embedding_provider_id` | Embedding Provider ID（必填） | "" |
| `auto_remember` | 是否自动记忆对话中的重要信息 | true |
| `memory_threshold` | 自动记忆的对话轮数阈值 | 5 |
| `top_k` | 检索时返回的记忆数量 | 5 |
| `embedding_cache_size` | Embedding 缓存大小，设为 0 禁用缓存 | 500 |
| `user_identity_map` | 用户身份映射配置（列表格式） | [] |
| `masters` | 主人列表，拥有最高权限（可查看和删除所有记忆） | [] |

### 用户身份映射配置

`user_identity_map` 是一个字符串列表，每个元素格式为 `"规范用户ID=会话ID1,会话ID2"`。

**示例配置：**
```json
{
  "user_identity_map": [
    "user001=qq_12345,wx_abcde",
    "user002=qq_67890,wx_fghij"
  ],
  "masters": ["user001", "user002"]
}
```

上述配置表示：
- `qq_12345` 和 `wx_abcde` 都是 `user001` 的身份
- `qq_67890` 和 `wx_fghij` 都是 `user002` 的身份
- `user001` 和 `user002` 是主人，拥有最高权限

## 🔐 记忆可见性分层

| 可见性 | 说明 | 适用场景 |
|--------|------|----------|
| `public` | 公共记忆，所有用户可见 | 家庭成员信息、基本设定 |
| `private` | 用户专属记忆，仅该用户可见 | 个人生日、个人偏好 |
| `secret` | 秘密记忆，仅主人可见 | 敏感凭证、隐私信息 |

### 权限规则

- **主人**: 可以查看和删除所有记忆
- **普通用户**: 只能查看公共记忆和自己的私有记忆
- **公共记忆**: 只有主人可以删除
- **私有/秘密记忆**: 只有所有者可以删除

## 🚫 智能去重机制

从 v1.3.0 开始，记忆系统支持智能去重：

- **去重阈值**: 相似度 ≥ 95% 的记忆会被判定为重复
- **去重范围**: 检查同一用户的记忆，以及公共记忆
- **处理方式**: 跳过重复记忆，返回已存在记忆的ID
- **日志记录**: 在日志中记录去重信息，方便调试

**去重效果示例：**
```
用户: 记住我的生日是3月15日
AI: ✓ 已记住: 我的生日是3月15日 (ID: 123, 类别: personal, 可见性: private)

用户: 记住我的生日是3月15日  (重复添加)
AI: ⚠️ 检测到重复记忆，已跳过。相似记忆 ID: 123
```

## ⚡ Embedding 缓存

从 v1.3.1 开始，新增 Embedding 缓存功能：

- **缓存策略**: LRU (Least Recently Used)
- **默认容量**: 500 条
- **缓存键**: 文本 MD5 哈希
- **效果**: 重复查询直接返回缓存结果，无需调用 API
- **预热机制**: 启动时自动预热 L0/L1 层记忆的 embedding

**缓存统计示例：**
```
📦 Embedding 缓存:
  • 缓存大小: 12/500
  • 命中次数: 45
  • 未命中次数: 12
  • 命中率: 78.9%
```

## 📁 数据存储

## 🔧 技术架构

```
用户消息 → 关键词提取 → L1 层预加载
                ↓
        Embedding(缓存) → L2 层向量搜索
                ↓
        L0 层始终加载
                ↓
        权限过滤 → 返回相关记忆
                ↓
        注入到 LLM 上下文
```

### 分层加载流程

```python
def get_context_memories(user_input: str) -> str:
    # 1. 始终加载 L0
    context = get_layer_memories("L0")
    
    # 2. 关键词匹配 L1
    keywords = extract_keywords(user_input)
    context += get_layer_memories("L1", keywords)
    
    # 3. 向量检索 L2 (top_k=3)
    context += vector_search(user_input, layer="L2", top_k=3)
    
    return context
```

### 为什么选择 ChromaDB？

- ✅ **无需额外部署** - 嵌入式运行，像 SQLite 一样简单
- ✅ **自动向量索引** - 使用 HNSW 算法，检索速度快
- ✅ **支持元数据过滤** - 可按类别、重要性、可见性、层级等条件筛选
- ✅ **Python 原生** - API 简洁，易于集成
